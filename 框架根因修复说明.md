# FDRE-HRDC 框架根因修复说明

## 1. 当前问题判断

本轮复盘后，我们不再把问题简单表述为“效果不好”，而是拆成三个可修复的工程问题：第一，原版 PPO 参数与 LunarLander-v3 不匹配，同一奖励函数在不同 PPO seed 下分数波动很大，说明训练器本身引入了额外方差；第二，LLM 对 LunarLander 可能存在训练数据先验，部分 seed 第 0 轮直接生成较好奖励，这不能完全证明搜索框架本身的能力；第三，失败 seed 中出现 `original_reward * 0.001` 这类 anchor 过度缩放，后续 16 轮局部修改都没有触及根因，说明原框架缺少结构性诊断、重启和跨 seed 共享机制。

## 2. 已完成的代码修复

已将 PPO 默认参数改为更接近 RL Zoo 的 LunarLander-v3 配置：`learning_rate=2.5e-4`、`n_steps=1024`、`n_epochs=4`、`gamma=0.999`、`gae_lambda=0.98`、`ent_coef=0.01`。这一步用于降低“同一奖励函数由于 PPO 配置不稳导致完全不同得分”的干扰。

已新增 `diagnostics.py`，对候选奖励函数做 anchor 静态检查。现在如果 LLM 生成 `original_reward * 0.001`、`anchor = 0.001; return anchor * original_reward` 或完全不用 `original_reward`，框架会在训练前判定为无效，并要求重新生成或修复。默认要求 `original_reward` 缩放系数不低于 `0.5`，推荐形式为 `reward = original_reward + shaping_terms`。

已在 prompt 中加入硬约束：不得将 `original_reward` 缩放到 0.5 以下，refine 阶段如果发现当前代码存在 tiny anchor，必须优先修复 anchor，而不是只调 shaping 项权重。

已加入结构性重启机制。当连续若干轮得分低于阈值时，不再强制从低质量 best_code 做保守 rollback，而是触发 fresh restart；只有历史 best 达到 `rollback_min_score` 以上时才允许保守回退。这样可以避免 seed_46 这类情况被锁死在一个错误局部邻域中。

已加入跨 seed 共享 reward pool。主 FDRE 多 seed 运行时，任一 seed 搜索到超过阈值的 reward，会写入共享池；后续失败 seed 在触发重启时可以从共享池读取高分候选作为新起点。这使实验从“多个 seed 各自随机碰运气”变为“多 seed 协同搜索”。

已将 LunarLander 配置中的任务描述改为黑盒任务描述，不再写入“平稳降落、保持直立、双腿接触、燃料”等人工领域提示，减少 LLM 直接调用训练先验的空间。

## 3. 建议复验方式

建议使用新增配置 `configs/lunarlander_robust_1m.json` 进行正式复验：

```powershell
python scripts/run_experiment.py --config configs/lunarlander_robust_1m.json --llm-provider ollama --llm-model qwen2.5-coder:7b --suite
```

该配置使用 10 seeds、1M timesteps、16 轮搜索、RL Zoo 风格 PPO 参数、anchor 硬检查、结构性重启和共享 reward pool。复验时建议重点记录每个 seed 的 `best_score`、`best_iter`、anchor 检查结果、是否触发 restart、是否使用 shared pool。

## 4. 对外解释口径

可以这样解释：前一版实验已经证明了“LLM 生成奖励 + PPO 训练 + 反馈迭代”这条路线有潜力，但根因分析表明原框架仍存在三类不足：训练器超参方差、LLM 环境先验干扰、失败 seed 缺乏结构性自救。当前修复版不是继续包装结果，而是把这些问题机制化解决：用 RL Zoo 参数稳定训练，用 anchor 静态诊断约束奖励目标不被削弱，用结构性重启跳出错误局部最优，用跨 seed 共享池把单 seed 随机搜索升级为协同搜索。因此，新版本更接近“自动化奖励函数搜索框架”，也更适合支撑论文里的方法论描述。
